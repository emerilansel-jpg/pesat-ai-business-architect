# Design — Pesat AI Business Architect v5.5 Fixes + Deployment Cleanup

**Date:** 2026-07-15  
**Status:** Approved  
**Scope:** Fix UI/ActivityPanel, stuck “crafting response” status, first-response main-prompt alignment, and clean up deployment/proxy issues.

## Goals

1. Make `ActivityPanel` feel like a fun mini-game with a walking minion, dynamic stages, and playful Indonesian status messages.
2. Eliminate the “stuck on crafting response” bug by ensuring every success/error path finishes the activity log and adding a network timeout guard.
3. Ensure the first AI response follows the full main prompt identity/flow while still focusing on greeting + data gathering.
4. Fix deployment inconsistencies:
   - health-check port mismatch (3000 → 3002)
   - remove hardcoded API keys from source
   - robust Caddy `/api` routing
   - refresh stale `COLDSTART.md`

## Section 1 — ActivityPanel “Game Mode”

### Context
Current `ActivityPanel` already has a minion, typewriter logs, progress bar, and fun facts. The user wants it to be *more* entertaining and game-like.

### Changes

#### `src/contexts/ActivityContext.tsx`
- Add `currentStage: ActivityStage` to provider state where:  
  `ActivityStage = 'idle' | 'thinking' | 'searching' | 'analyzing' | 'crafting' | 'success' | 'error'`
- Expose `setStage(stage: ActivityStage)`.
- Keep existing `logs`, `isVisible`, `isProcessing`, typewriter logic.
- Logs remain for textual history; stage drives the visual animation.

#### `src/components/ActivityPanel.tsx`
- Map each stage to minion animation state:
  - `thinking`: slow walk.
  - `searching` / `analyzing`: faster walk.
  - `crafting`: typing on laptop.
  - `success`: jump + wave + confetti burst.
  - `error`: confused/sad + “Coba lagi?” hint.
- Progress bar becomes an “XP bar”:
  - `thinking` → 15%
  - `searching` → 35%
  - `analyzing` → 55%
  - `crafting` → 80%
  - `success` → 100%
- Speech bubble with rotating fun messages per stage, e.g.:
  - “AI-nya lagi ngopi dulu ☕”
  - “Lagi baca bisnis kamu sampai ke akar-akarnya 🔍”
  - “Sedang rakit jawaban yang bikin ‘wah’ ✨”
  - “Selesai! Quest complete 🏆”
- On success show a “Quest complete!” badge and brief fun fact.
- Keep desktop right-side panel and mobile bottom sheet behavior.

## Section 2 — Fix “stuck crafting response”

### Context
`App.tsx` calls `addLog(ACTIVITY.crafting, 'sparkle')` before `sendMessage(...)`. On success it calls `markLastDone()` then `finishLog(ACTIVITY.success, ...)`. On error it calls `finishLog(ACTIVITY.error, ...)`.

### Changes

#### `src/App.tsx`
- Add an `AbortController` with a 60-second timeout around `sendMessage(...)` so hung requests always transition to the `catch` block.
- In `catch`, call `markLastDone()` before `finishLog(ACTIVITY.error, 'error')` so no log remains in `typing` state.
- Ensure `finally` always clears `isProcessingRef` and `setProcessing(false)`.

## Section 3 — First Response Aligns with Main Prompt

### Context
The first user message currently uses only `settings.stepPrompts[0].prompt`, which includes the base persona but lacks the full 7-stage flow and detailed guidance from `MAIN_SYSTEM_PROMPT`.

### Changes

#### `src/App.tsx`
Replace the first-message system prompt construction with:

```ts
const firstMessageSystem = isFirstUserMessage
  ? MAIN_SYSTEM_PROMPT +
    '\n\n' +
    step1FocusedInstructions +
    (inlineVizPrompt || '') +
    searchContext +
    choicePrompt
  : MAIN_SYSTEM_PROMPT +
    (inlineVizPrompt || '') +
    searchContext +
    choicePrompt;
```

Where `step1FocusedInstructions` is the `## GOAL`, `## TASKS`, `## FORMAT`, and `## PEMBUKAAN DEFAULT` from Step 1.

This keeps the greeting/data-gathering focus while embedding the full identity and flow.

## Section 4 — Deployment Cleanup

### 4.1 Fix port mismatch

#### `scripts/deploy.js`
Change health check from:
```js
const health = await ssh.execCommand('curl -s http://localhost:3000/health');
```
to:
```js
const health = await ssh.execCommand('curl -s http://localhost:3002/health');
```

#### `scripts/restart-proxy.js`
Same change: `localhost:3000/health` → `localhost:3002/health`.

### 4.2 Remove hardcoded API keys

#### `scripts/advisor-proxy.js`
- Remove `const OPENAI_KEY = '...'`.
- Read keys from environment variables:
  - `OPENAI_API_KEY`
  - `DEEPSEEK_API_KEY`
  - `TAVILY_API_KEY`
- Use request body `apiKey` only as fallback, e.g.:
  ```js
  const apiKey = data.apiKey || process.env.OPENAI_API_KEY;
  if (!apiKey) { ... return 400 ... }
  ```

#### `.env` (new, gitignored)
Create a `.env` file at the project root containing the current production keys. It will be uploaded to the VPS and sourced by PM2.

#### `src/services/settings.ts`
- Set `openaiKey`, `deepseekKey`, `tavilyKey` default values to empty strings.
- Keep the admin UI able to override; if admin leaves them empty, the proxy uses its environment keys.

### 4.3 Robust Caddy config

#### `scripts/final-deploy.js`
- Generate Caddy snippet using `handle_path /api/*` instead of `handle /api/*` so `/api/chat` is forwarded to the backend as `/chat`.
- Detect the Docker host gateway dynamically:
  ```bash
  ip route | awk '/default/ {print $3}'
  ```
  inside the Caddy container, and use that IP for `reverse_proxy`.
- Validate the Caddyfile before restarting:
  ```bash
  docker exec pesat-control-plane-caddy-1 caddy validate --config /etc/caddy/Caddyfile --adapter caddyfile
  ```

### 4.4 Refresh `COLDSTART.md`

Update to reflect the actual current state:
- Router: `BrowserRouter` with basename `/advisor`.
- Image provider: `openai-image` (gpt-image-1) with `pollinations` fallback.
- Architecture: Cloudflare tunnel → Caddy (pesat-control-plane) → static builds + proxy.
- Proxy port: `3002`.
- VPS IP: `148.230.103.98`.
- Remove old root password and old nginx/IP references.
- Add `ActivityPanel` and `ActivityContext` to file structure.
- Add note about `.env` and PM2.
